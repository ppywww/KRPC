# Krpc 项目开发过程详细记录

> 本文档记录了从零开始开发 Krpc C++ RPC 框架的完整过程，包括每一步的设计决策和原因分析。

---

## 阶段一：需求分析与架构设计

### 1.1 为什么需要 RPC 框架？

**问题背景**：
在分布式系统中，服务之间需要通信。如果使用 HTTP REST API，每次调用都需要：
1. 手动序列化/反序列化参数和返回值
2. 处理网络连接管理
3. 处理各种网络错误
4. 无法直接调用远程方法，像调用本地函数一样

**RPC (Remote Procedure Call) 解决**：
- 让远程调用看起来像本地调用
- 自动处理序列化、网络通信、错误处理

### 1.2 技术选型决策

| 组件 | 选择 | 原因 |
|------|------|------|
| **网络库** | Muduo | 高性能 Reactor 模型，已封装 epoll，开箱即用 |
| **序列化** | Protobuf | 高效二进制序列化，跨语言支持，Google 背书 |
| **服务发现** | ZooKeeper | 成熟稳定的分布式协调服务 |
| **日志** | glog | Google 标准日志库，功能完善 |
| **C++ 标准** | C++11 | 足够现代，支持智能指针、lambda、线程库 |

### 1.3 架构设计

```
┌─────────────────────────────────────────────────────────────────┐
│                         客户端 (Caller)                          │
│  ┌─────────────────┐    ┌─────────────────┐                    │
│  │  UserCode       │───▶│  Stub (Proxy)   │───▶ KrpcChannel    │
│  │  (业务代码)      │    │  (RPC代理)       │    (网络通信)      │
│  └─────────────────┘    └─────────────────┘                    │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ TCP/IP
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                         服务端 (Callee)                          │
│  ┌─────────────────┐    ┌─────────────────┐                    │
│  │  KrpcProvider   │◀───│  Service       │◀─── UserCode       │
│  │  (网络框架)      │    │  (业务服务)     │    (业务代码)       │
│  └─────────────────┘    └─────────────────┘                    │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    ┌─────────────────┐
                    │   ZooKeeper     │
                    │   (服务注册)     │
                    └─────────────────┘
```

**设计原因**：
1. **分离关注点**：网络通信和业务逻辑完全解耦
2. **统一入口**：KrpcChannel 统一处理所有网络通信
3. **服务注册**：ZooKeeper 实现服务发现，解耦服务提供者和消费者

## 阶段三：定义 RPC 协议

### 3.1 为什么需要自定义协议？

TCP 是流式协议，存在**粘包**和**拆包**问题：

```
发送端:                    接收端:
send("Hello") ────┐        recv() ──── 可能收到 "HelloWorld"
send("World")  ──┘           或 "Hel" + "loWorld"
```

**解决方案**：自定义应用层协议，在数据前加长度头

### 3.2 创建 Krpcheader.proto

```protobuf
syntax = "proto3";
package Krpc;

message RpcHeader {
    bytes service_name = 1;  // 服务名
    bytes method_name = 2;    // 方法名
    uint32 args_size = 3;     // 参数长度
}
```

**设计原因**：
- `service_name`: 用于 ZooKeeper 服务发现
- `method_name`: 指定调用哪个方法
- `args_size`: 解决粘包问题，知道参数有多长

### 3.3 最终协议格式

```
┌─────────────────────────────────────────────────────────────┐
│ [4B Total Len] [4B Header Len] [Header Data] [Args Data]  │
├──────────────┬──────────────┬──────────────┬────────────────┤
│  总长度(4B)  │ 头部长度(4B) │   Protobuf    │  序列化参数    │
│  (uint32)    │  (uint32)   │   头部        |  (二进制)      │
└──────────────┴──────────────┴──────────────┴────────────────┘
```

**协议解析流程**：
1. 先读 4 字节，得到 Total Length
2. 再读 4 字节，得到 Header Length
3. 根据 Header Length 读取 Header
4. 根据 Args_Size 读取 Args

## 阶段四：配置模块开发

### 4.1 为什么需要配置模块？

RPC 框架需要很多配置：
- 服务端 IP 和端口
- ZooKeeper 地址
- 日志目录

硬编码不利于部署和修改，需要配置文件。

### 4.2 Krpcconfig.h 设计

```cpp
#ifndef _Krpcconfig_h
#define _Krpcconfig_h
#include <unordered_map>
#include <string>

class Krpcconfig {
public:
    void LoadConfigFile(const char *config_file);  // 加载配置文件
    std::string Load(const std::string &key);      // 根据 key 获取 value

    std::string GetLogDir() {
        return Load("log_dir");  // 快捷方法获取日志目录
    }

private:
    std::unordered_map<std::string, std::string> config_map;  // 存储配置
    void Trim(std::string &read_buf);  // 去除字符串首尾空格
};
#endif
```

**设计决策**：
- `unordered_map`: O(1) 查询性能，比 map 更快
- `Trim()`: 处理配置文件中的空格，提高健壮性
- `GetLogDir()`: 提供便捷方法，避免每次调用 Load()

### 4.3 Krpcconfig.cc 实现要点

```cpp
void Krpcconfig::LoadConfigFile(const char *config_file) {
    // 1. 打开配置文件
    std::unique_ptr<FILE, FileCloser> pf(fopen(config_file, "r"), FileCloser());

    // 2. 逐行读取
    while (fgets(buf, 1024, pf.get()) != nullptr) {
        std::string read_buf(buf);
        Trim(read_buf);

        // 3. 跳过注释和空行
        if (read_buf[0] == '#' || read_buf.empty()) continue;

        // 4. 解析 key=value
        int index = read_buf.find('=');
        if (index == -1) continue;

        std::string key = read_buf.substr(0, index);
        std::string value = read_buf.substr(index + 1);
        Trim(key);
        Trim(value);

        // 5. 存入 map
        config_map.insert({key, value});
    }
}
```

**关键点说明**：
- `unique_ptr`: 自动管理文件句柄，防止资源泄漏
- `fgets`: 按行读取，比 `fscanf` 更安全
- `find('=')`: 简单的 key=value 解析

### 5.2 KrpcLogger.h 设计

```cpp
class KrpcLogger {
public:
    // 构造函数，自动初始化 glog
    explicit KrpcLogger(const char *argv0, const std::string& log_dir = "") {
        google::InitGoogleLogging(argv0);

        if (!log_dir.empty()) {
            mkdir(log_dir.c_str(), 0755);  // 创建日志目录
            FLAGS_log_dir = log_dir;
            FLAGS_logtostderr = false;       // 只输出到文件
        } else {
            FLAGS_logtostderr = true;        // 输出到控制台
        }

        FLAGS_colorlogtostderr = true;
        FLAGS_minloglevel = 0;
        FLAGS_max_log_size = 100;
    }

    ~KrpcLogger() {
        google::ShutdownGoogleLogging();
    }

    // 静态日志方法
    static void Info(const std::string &message) { LOG(INFO) << message; }
    static void Warning(const std::string &message) { LOG(WARNING) << message; }
    static void ERROR(const std::string &message) { LOG(ERROR) << message; }
};
```

**设计决策**：
- **RAII 手法**: 构造函数初始化，析构函数清理，确保日志系统生命周期正确
- **静态方法**: 不需要创建实例就能记录日志

## 阶段七：ZooKeeper 客户端封装

### 7.1 为什么需要 ZooKeeper？

在分布式系统中，服务消费者需要知道服务提供者的地址：
- **硬编码问题**: 服务者地址变化时，所有消费者都要改
- **ZooKeeper 解决方案**: 服务者注册到 ZooKeeper，消费者从 ZooKeeper 查询

```
服务端启动 ──▶ 注册服务到 ZooKeeper ──▶ 创建临时节点
                                     ──▶ 节点数据: ip:port

客户端启动 ──▶ 查询 ZooKeeper ──▶ 获取服务地址 ──▶ 建立连接
```

### 7.2 ZkClient.h 设计

```cpp
class ZkClient {
public:
    ZkClient();
    ~ZkClient();

    void Start();  // 连接 ZooKeeper 服务器
    void Create(const char* path, const char* data, int datalen, int state = 0);
    std::string GetData(const char* path);

private:
    zhandle_t* m_zhandle;  // ZooKeeper 句柄
};
```

**设计决策**：
- `zhandle_t`: ZooKeeper 的核心句柄，所有操作都依赖它
- `Create()`: 创建节点，state 参数区分永久/临时节点
- `GetData()`: 获取节点数据（包含服务地址）

### 7.3 Start() 函数实现

```cpp
void ZkClient::Start() {
    // 1. 从配置获取 ZooKeeper 地址
    std::string host = KrpcApplication::GetInstance().GetConfig().Load("zookeeperip");
    std::string port = KrpcApplication::GetInstance().GetConfig().Load("zookeeperport");
    std::string connstr = host + ":" + port;

    // 2. 初始化连接
    // zookeeper_mt: 多线程版本，使用三个线程处理 I/O 和回调
    m_zhandle = zookeeper_init(connstr.c_str(), global_watcher, 6000, nullptr, nullptr, 0);

    // 3. 等待连接成功（通过全局 watcher 回调通知）
    std::unique_lock<std::mutex> lock(cv_mutex);
    cv.wait(lock, [] { return is_connected; });
}
```

**关键点**：
- `zookeeper_init`: 异步建立连接，立即返回
- `global_watcher`: 全局观察器，连接成功时会被调用
- `cv.wait`: 等待连接建立，避免后续操作在未连接时执行

### 7.4 Create() 函数实现

```cpp
void ZkClient::Create(const char* path, const char* data, int datalen, int state) {
    // ZOO_EPHEMERAL: 临时节点，客户端断开连接后自动删除
    // 这对于服务发现很重要：服务宕机后，节点自动消失
    flag = zoo_create(m_zhandle, path, data, datalen,
                      &ZOO_OPEN_ACL_UNSAFE, state, path_buffer, bufferlen);
}
```

**临时节点的意义**：
- 服务崩溃时，TCP 连接断开 → ZooKeeper 检测到 → 自动删除临时节点
- 客户端立即感知服务不可用，无需心跳检测

## 阶段九：服务提供者开发

### 9.1 为什么需要 KrpcProvider？

服务端需要：
1. 接收网络请求
2. 解析请求数据
3. 调用对应的服务方法
4. 返回结果

这些都需要一个统一的服务提供者来管理。

### 9.2 KrpcProvider.h 设计

```cpp
class KrpcProvider {
public:
    void NotifyService(google::protobuf::Service* service);  // 发布服务
    void Run();  // 启动 RPC 服务

private:
    struct ServiceInfo {
        google::protobuf::Service* service;
        std::unordered_map<std::string, const google::protobuf::MethodDescriptor*> method_map;
    };

    std::unordered_map<std::string, ServiceInfo> service_map;
    muduo::net::EventLoop event_loop;  // Muduo 事件循环

    void OnConnection(const muduo::net::TcpConnectionPtr& conn);
    void OnMessage(const muduo::net::TcpConnectionPtr& conn, muduo::net::Buffer* buffer, muduo::Timestamp receive_time);
    void SendRpcResponse(const muduo::net::TcpConnectionPtr& conn, google::protobuf::Message* response);
};
```

**设计决策**：
- `NotifyService`: 发布服务，参数是 `google::protobuf::Service*`，利用多态
- `service_map`: 存储所有已发布的服务和方法
- `EventLoop`: Muduo 的事件循环，基于 Reactor 模型

### 9.3 NotifyService() 实现

```cpp
void KrpcProvider::NotifyService(google::protobuf::Service* service) {
    // 1. 获取服务描述符
    const google::protobuf::ServiceDescriptor *psd = service->GetDescriptor();

    // 2. 获取服务名
    std::string service_name = psd->name();

    // 3. 遍历所有方法
    for (int i = 0; i < method_count; ++i) {
        const google::protobuf::MethodDescriptor *pmd = psd->method(i);
        std::string method_name = pmd->name();
        service_info.method_map.emplace(method_name, pmd);
    }

    // 4. 保存到 map
    service_map.emplace(service_name, service_info);
}
```

**设计原因**：
- `GetDescriptor()`: Protobuf 生成的代码包含服务描述信息
- `method_count + method(i)`: 遍历获取所有方法
- **动态多态**: 基类指针指向子类对象，调用子类的重写方法

### 9.4 Run() 函数实现

```cpp
void KrpcProvider::Run() {
    // 1. 从配置读取 IP 和端口
    std::string ip = KrpcApplication::GetInstance().GetConfig().Load("rpcserverip");
    int port = atoi(KrpcApplication::GetInstance().GetConfig().Load("rpcserverport").c_str());

    // 2. 创建 TcpServer
    muduo::net::InetAddress address(ip, port);
    std::shared_ptr<muduo::net::TcpServer> server =
        std::make_shared<muduo::net::TcpServer>(&event_loop, address, "KrpcProvider");

    // 3. 绑定回调
    server->setConnectionCallback(std::bind(&KrpcProvider::OnConnection, ...));
    server->setMessageCallback(std::bind(&KrpcProvider::OnMessage, ...));

    // 4. 注册到 ZooKeeper
    ZkClient zkclient;
    zkclient.Start();
    for (auto &sp : service_map) {
        std::string service_path = "/" + sp.first;
        zkclient.Create(service_path.c_str(), nullptr, 0);
        for (auto &mp : sp.second.method_map) {
            std::string method_path = service_path + "/" + mp.first;
            sprintf(method_path_data, "%s:%d", ip.c_str(), port);
            zkclient.Create(method_path.c_str(), method_path_data, strlen(method_path_data), ZOO_EPHEMERAL);
        }
    }

    // 5. 启动服务
    server->start();
    event_loop.loop();
}
```

**关键步骤说明**：
1. **Muduo TcpServer**: 基于 Reactor 模式，高性能网络通信
2. **回调绑定**: `setConnectionCallback` 处理连接，`setMessageCallback` 处理消息
3. **ZooKeeper 注册**: 服务启动时注册，消费者可发现

### 9.5 OnMessage() 消息处理（核心）

```cpp
void KrpcProvider::OnMessage(const muduo::net::TcpConnectionPtr& conn,
                              muduo::net::Buffer* buffer,
                              muduo::Timestamp receive_time) {
    // 1. 解决粘包：循环处理直到数据不足
    while (buffer->readableBytes() >= 4) {
        // 2. 读取总长度
        uint32_t total_len = 0;
        std::memcpy(&total_len, buffer->peek(), 4);
        total_len = ntohl(total_len);

        // 3. 检查数据是否完整
        if (buffer->readableBytes() < 4 + total_len) {
            break;  // 数据不完整，等待更多数据
        }

        // 4. 消费长度头
        buffer->retrieve(4);

        // 5. 读取 Header
        uint32_t header_len = 0;
        std::memcpy(&header_len, buffer->peek(), 4);
        header_len = ntohl(header_len);
        buffer->retrieve(4);

        std::string rpc_header_str(buffer->peek(), header_len);
        Krpc::RpcHeader krpcHeader;
        krpcHeader.ParseFromString(rpc_header_str);
        buffer->retrieve(header_len);

        // 6. 读取 Body
        uint32_t args_size = total_len - 4 - header_len;
        std::string args_str(buffer->peek(), args_size);
        buffer->retrieve(args_size);

        // 7. 获取服务和方法
        std::string service_name = krpcHeader.service_name();
        std::string method_name = krpcHeader.method_name();

        // 8. 查找服务
        auto it = service_map.find(service_name);
        if (it == service_map.end()) { /* 处理错误 */ }

        // 9. 获取方法描述符
        auto mit = it->second.method_map.find(method_name);
        const google::protobuf::MethodDescriptor* method = mit->second;

        // 10. 获取服务对象
        google::protobuf::Service* service = it->second.service;

        // 11. 创建请求和响应对象
        const google::protobuf::Message* request = service->GetRequestPrototype(method).New();
        google::protobuf::Message* response = service->GetResponsePrototype(method).New();

        // 12. 反序列化请求参数
        request->ParseFromString(args_str);

        // 13. 调用服务方法 (在 service 上调用)
        // 这里的 CallMethod 是 protobuf 提供的，会调用子类的重写方法
        service->CallMethod(method, nullptr, request, response, nullptr);

        // 14. 发送响应
        SendRpcResponse(conn, response);

        delete request;
        delete response;
    }
}
```

## 阶段十：RPC Channel 开发

### 10.1 为什么需要 KrpcChannel？

客户端调用 RPC 时，需要：
1. 查询 ZooKeeper 获取服务地址
2. 建立 TCP 连接
3. 序列化请求并发送
4. 接收并反序列化响应

KrpcChannel 继承 `google::protobuf::RpcChannel`，是 Protobuf 规定的客户端桩接口。

### 10.2 KrpcChannel.h 设计

```cpp
class KrpcChannel : public google::protobuf::RpcChannel {
public:
    KrpcChannel(bool connectNow);
    virtual ~KrpcChannel() { close(m_clientfd); }

    // Protobuf 规定的接口，调用远程方法
    void CallMethod(const ::google::protobuf::MethodDescriptor *method,
                    ::google::protobuf::RpcController *controller,
                    const ::google::protobuf::Message *request,
                    ::google::protobuf::Message *response,
                    ::google::protobuf::Closure *done) override;

private:
    int m_clientfd;
    std::string service_name;
    std::string m_ip;
    uint16_t m_port;
    std::string method_name;
    int m_idx;  // ip:port 分隔符位置

    bool newConnect(const char *ip, uint16_t port);
    std::string QueryServiceHost(ZkClient *zkclient, std::string service_name,
                                 std::string method_name, int &idx);
    ssize_t recv_exact(int fd, char* buf, size_t size);  // 确保读取指定字节
};
```

**设计原因**：
- **继承 RpcChannel**: Protobuf 要求，实现标准接口后可以自动生成 stub
- **m_clientfd**: 保存 socket 句柄，避免重复连接
- **recv_exact**: 确保读取完整数据，解决粘包问题

### 10.3 CallMethod() 实现（核心）

```cpp
void KrpcChannel::CallMethod(...) {
    // 1. 如果未连接，先连接
    if (-1 == m_clientfd) {
        // 1.1 查询 ZooKeeper 获取服务地址
        ZkClient zkCli;
        zkCli.Start();
        std::string host_data = QueryServiceHost(&zkCli, service_name, method_name, m_idx);

        // 1.2 解析 ip:port
        m_ip = host_data.substr(0, m_idx);
        m_port = atoi(host_data.substr(m_idx + 1).c_str());

        // 1.3 建立连接
        newConnect(m_ip.c_str(), m_port);
    }

    // 2. 序列化请求
    std::string args_str;
    request->SerializeToString(&args_str);

    // 3. 构建协议头
    Krpc::RpcHeader krpcheader;
    krpcheader.set_service_name(service_name);
    krpcheader.set_method_name(method_name);
    krpcheader.set_args_size(args_str.size());
    std::string rpc_header_str;
    krpcheader.SerializeToString(&rpc_header_str);

    // 4. 打包数据发送
    // 格式: [4B Total Len][4B Header Len][Header][Args]
    uint32_t header_size = rpc_header_str.size();
    uint32_t total_len = 4 + header_size + args_str.size();

    uint32_t net_total_len = htonl(total_len);
    uint32_t net_header_len = htonl(header_size);

    std::string send_rpc_str;
    send_rpc_str.append((char*)&net_total_len, 4);
    send_rpc_str.append((char*)&net_header_len, 4);
    send_rpc_str.append(rpc_header_str);
    send_rpc_str.append(args_str);

    send(m_clientfd, send_rpc_str.c_str(), send_rpc_str.size(), 0);

    // 5. 接收响应
    uint32_t response_len = 0;
    recv_exact(m_clientfd, (char*)&response_len, 4);  // 先读长度
    response_len = ntohl(response_len);

    std::vector<char> recv_buf(response_len);
    recv_exact(m_clientfd, recv_buf.data(), response_len);  // 再读内容

    // 6. 反序列化响应
    response->ParseFromArray(recv_buf.data(), response_len);
}
```

### 10.4 QueryServiceHost() 实现

```cpp
std::string KrpcChannel::QueryServiceHost(ZkClient *zkclient,
                                          std::string service_name,
                                          std::string method_name,
                                          int &idx) {
    // ZooKeeper 节点路径: /ServiceName/MethodName
    std::string path = "/" + service_name + "/" + method_name;
    std::string data = zkclient->GetData(path.c_str());  // 获取 ip:port

    // 找到冒号位置，用于分割 ip 和 port
    idx = data.find(':');
    return data;
}
```

**调用链**：
```
stub.Login() ──▶ KrpcChannel.CallMethod() ──▶ ZooKeeper 查询 ──▶ TCP 发送
                     │                                        │
                     ◀── TCP 接收 ──▶ 反序列化响应 ◀──────────┘
```

## 阶段十一：示例代码开发

### 11.1 定义 user.proto

```protobuf
syntax = "proto3";
package Kuser;

option cc_generic_services = true;

message ResultCode {
    int32 errcode = 1;
    bytes errmsg = 2;
}

message LoginRequest {
    bytes name = 1;
    bytes pwd = 2;
}

message LoginResponse {
    ResultCode result = 1;
    bool success = 2;
}

service UserServiceRpc {
    rpc Login(LoginRequest) returns (LoginResponse);
}
```

**关键点**：
- `option cc_generic_services = true`: 生成 C++ 服务类
- `service UserServiceRpc`: 定义 RPC 服务
- Protobuf 会自动生成 `UserServiceRpc_Stub` 类

### 11.2 服务端 Kserver.cc

```cpp
// 1. 定义本地服务类，继承 Protobuf 生成的服务基类
class UserService : public Kuser::UserServiceRpc {
public:
    // 2. 实现本地业务逻辑
    bool Login(std::string name, std::string pwd) {
        std::cout << "本地登录: name=" << name << " pwd=" << pwd << std::endl;
        return true;
    }

    // 3. 重写 RPC 方法 (由框架调用)
    void Login(::google::protobuf::RpcController* controller,
               const ::Kuser::LoginRequest* request,
               ::Kuser::LoginResponse* response,
               ::google::protobuf::Closure* done) override {
        // 从请求获取参数
        std::string name = request->name();
        std::string pwd = request->pwd();

        // 调用本地业务
        bool login_result = Login(name, pwd);

        // 填充响应
        response->mutable_result()->set_errcode(0);
        response->set_success(login_result);

        // 4. 执行回调 (框架会发送响应)
        done->Run();
    }
};

int main(int argc, char** argv) {
    // 5. 初始化框架
    KrpcApplication::Init(argc, argv);

    // 6. 创建服务提供者
    KrpcProvider provider;

    // 7. 发布服务
    provider.NotifyService(new UserService());

    // 8. 启动服务
    provider.Run();

    return 0;
}
```

**执行流程**：
```
收到 RPC 请求
    │
    ▼
KrpcProvider::OnMessage()
    │
    ▼
反序列化请求 → 找到 UserService::Login()
    │
    ▼
调用 Login() →执行业务逻辑
    │
    ▼
填充 LoginResponse
    │
    ▼
done->Run() → 发送响应给客户端
```

### 11.3 客户端 Kclient.cc

```cpp
int main(int argc, char** argv) {
    // 1. 初始化框架
    KrpcApplication::Init(argc, argv);

    // 2. 创建 RPC Stub (需要传入 KrpcChannel)
    Kuser::UserServiceRpc_Stub stub(new KrpcChannel(false));

    // 3. 准备请求
    Kuser::LoginRequest request;
    request.set_name("zhangsan");
    request.set_pwd("123456");

    // 4. 准备响应
    Kuser::LoginResponse response;

    // 5. 创建控制器
    Krpccontroller controller;

    // 6. 调用 RPC
    stub.Login(&controller, &request, &response, nullptr);

    // 7. 检查结果
    if (controller.Failed()) {
        std::cout << "RPC 调用失败: " << controller.ErrorText() << std::endl;
    } else {
        if (response.result().errcode() == 0) {
            std::cout << "登录成功: " << response.success() << std::endl;
        }
    }

    return 0;
}
```

## 开发总结

### 核心设计模式

| 模式 | 应用场景 | 好处 |
|------|----------|------|
| **单例模式** | KrpcApplication | 全局唯一实例，节省资源 |
| **工厂模式** | protobuf service | 动态创建服务实例 |
| **代理模式** | KrpcChannel | 封装网络通信细节 |
| **观察者模式** | ZooKeeper watcher | 事件通知机制 |
| **策略模式** | RpcController | 可替换的错误处理策略 |

### 关键设计决策回顾

1. **为什么选择 Muduo？**
   - 已封装完善的 Reactor 模型
   - 高性能 epoll 使用
   - 减少网络编程复杂度

2. **为什么用 Protobuf？**
   - 高效二进制序列化
   - 自动生成代码
   - 跨语言支持

3. **为什么用 ZooKeeper？**
   - 成熟的服务注册/发现方案
   - 临时节点自动清理
   - Watcher 机制实时通知

4. **自定义协议的原因？**
   - 解决 TCP 粘包/拆包问题
   - 统一的 RPC 数据格式
   - 便于扩展和版本升级
